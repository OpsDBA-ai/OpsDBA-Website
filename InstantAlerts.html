Active Alert Scenarios

Storage Subsystem

  • Critical I/O Stalls (Error 833)

    Identifies storage subsystem saturation. When disk requests exceed 15s, we capture the specific file and database context immediately.

  • Read-Retry Warnings (Error 825)

    The single most important indicator of physical drive failure. SQL Server successfully read the data but had to "try again"â€”a clear sign of a dying disk.

  • Hard Corruption (Error 823/824)

    Real-time detection of physical IO errors or logical page checksum failures during disk operations.

Performance & Pressure

  • Sustained CPU Pressure (>90%)

    Monitoring of the Resource Pool. We alert when the engine is CPU-throttled for a sustained period, identifying runaway queries.

  • Memory Thrashing (PLE)

    Real-time Page Life Expectancy monitoring. If PLE drops below 300s, it signifies the engine is aggressively flushing data from RAM.

  • Deadlock Victim Capture (Error 1205)

    Instantly identifies processes killed by the engine due to cyclic dependencies for immediate concurrency tuning.

Operations & Security

  • Transaction Log Full (Error 9002)

    Prevents service-wide "Read-Only" freezes by alerting before storage exhaustion halts all activity.

  • Backup Engine Failure (Error 3041)

    Immediate notification when a native SQL backup fails, protecting your Recovery Point Objective (RPO).

  • Failed Logins (Error 18456)

    Monitoring of unauthorized access attempts to identify brute-force patterns or misconfigured accounts.

Severity & Exception Deep-Dive

Severity 17-19 Resource Pressure

Identifies bottlenecks including Insufficient Resources and Thread Starvation.

Severity 20-22 Fatal Engine

Signifies tasks terminated due to severe engine exceptions or logical corruption.

Severity 23-25 Fatal Hardware

Indicates Physical Media Failures or severe hardware inconsistencies.

Error 833 IO Stall Latency

Detection of I/O requests exceeding 15 seconds, identifying storage stalls.

Remediation First

OpsDBA InstantAlerts utilize internal SQL Agent Alerting in the msdb database. We use a proprietary responder that executes a local T-SQL Error Log scrape, ensuring the root cause is included in every outbound notification.
